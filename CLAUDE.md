# Votera - Project Rules

## Platform
- This app must work on **web, Android, and iOS**.
- Never use `dart:io` directly — use platform-agnostic alternatives.
- Test UI on both mobile and web viewports.

## Architecture
- Follow **Clean Architecture** with feature-based structure.
- Each feature has: `data/`, `domain/`, `presentation/`, `di/`.
- Each feature has its own DI file inside `feature/di/` — registered in `core/di/injection_container.dart`.
- State management: **BLoC / Cubit** with `flutter_bloc`.
- Routing: **go_router** (no code generation needed).
- Dependency injection: **get_it**.
- Network: **Dio** with `ApiClient` abstraction.
- Error handling: `dartz Either<Failure, T>` in repositories and use cases.

## Code Quality Rules
1. Write code using best practices and clean architecture principles.
   - Code must be efficient, maintainable, and scalable.

2. Write code in a human-readable way.
   - Logic should be clear and easy to understand.
   - Avoid clever or confusing implementations unless absolutely necessary.

3. Add clear, well-structured comments where needed.
   - Comments must explain *why* the code exists, not just what it does.
   - Do not use emojis or icons in comments.
   - Keep comments simple, professional, and easy to read.

4. Write functions as modular, independent building blocks.
   - Each function should have a single responsibility.
   - Functions must be easy to reuse, rearrange, or refactor if the flow changes.
   - Avoid tightly coupled logic.

5. General rules:
   - Prefer clarity over brevity.
   - Avoid unnecessary abstractions.
   - Follow consistent naming conventions.
   - Code should look like it was written by an experienced human developer, not autogenerated.

## Project Structure
```
lib/
  main.dart                    -- Single entry point
  bootstrap.dart               -- App initialization
  app/view/                    -- Root App widget, ShellPage
  core/
    config/                    -- AppConfig (single environment)
    constants/                 -- App-wide constants
    design_system/             -- Tokens, components, theme, utils
    di/                        -- Core injection container
    domain/services/           -- Core abstractions (AuthTokenProvider)
    error/                     -- Failures, Exceptions
    network/                   -- ApiClient, DioApiClient, AuthInterceptor, NetworkInfo
    router/                    -- GoRouter setup
    usecases/                  -- Base UseCase class
    utils/                     -- Shared utilities
  features/
    <feature>/
      data/
        datasources/           -- Remote and local data sources
        models/                -- Data models (fromJson/toJson)
        repositories/          -- Repository implementations
        services/              -- Feature-specific services
      domain/
        entities/              -- Business entities
        repositories/          -- Repository interfaces
        usecases/              -- Use cases
      presentation/
        bloc/                  -- BLoC / Cubit
        pages/                 -- Full screen pages
        widgets/               -- Feature-specific widgets
      di/                      -- Feature DI registration
  l10n/arb/                    -- Localization files
  shared/widgets/              -- Shared reusable widgets
```

## Dependencies
- `go_router` for routing
- `flutter_bloc` / `bloc` for state management
- `get_it` for DI
- `dio` + `pretty_dio_logger` for networking
- `dartz` for functional error handling
- `equatable` for value equality
- `flutter_secure_storage` for token storage
- `shared_preferences` for simple key-value storage
- `flutter_screenutil` for responsive sizing
- `flutter_svg` for SVG assets
- `google_fonts` for typography

## Conventions
- No emojis in code or comments.
- Use `const` constructors wherever possible.
- BLoC events and states use `part` files.
- Models extend domain entities.
- Repository implementations check `NetworkInfo.isConnected` before remote calls.
- All feature DI lives in `feature/di/<feature>_injection.dart`.
